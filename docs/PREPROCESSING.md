# Серверная конвертация LAS → COPC (PR3)

Инфраструктурный PR — без кода приложения. Установка PDAL/entwine на
сервере и проверенная вручную команда конвертации, на которую будет
опираться LAS-стриминг (PR4).

## Что установлено

Conda-окружение `geo` через [Miniforge](https://github.com/conda-forge/miniforge)
(установлен в `C:\Users\admin\miniforge3`, изолированно — не задействует
системный Python 3.13, который уже стоял на сервере и используется чем-то
другим):

```
C:\Users\admin\miniforge3\condabin\conda.bat create -n geo -c conda-forge pdal entwine -y
```

Бинарники: `C:\Users\admin\miniforge3\envs\geo\Library\bin\pdal.exe`
(проверено — версия 2.10.2) и `entwine.exe` (версия 3.2.1).

## Важное открытие: entwine НЕ нужен для нашего пайплайна

PDAL 2.10 умеет писать COPC **напрямую** (`writers.copc`, подтверждено
`pdal --drivers`) — однопроходная команда `pdal translate in.las
out.copc.laz` уже строит внутренний octree внутри одного `.laz`-файла.

Entwine решает другую задачу — объединение **нескольких** облётов/сканов
в один общий индексированный набор (EPT/COPC), с возможностью домержа
позже (`entwine merge`). У нас один тур = один LAS-файл, поэтому entwine
избыточен для текущего пайплайна. Установлен и доступен на будущее (если
когда-нибудь понадобится сводить несколько сканов одного объекта в один
индекс), но **в команду конвертации тура (PR4) entwine не входит** — это
сознательное упрощение, найденное при проверке, а не недосмотр.

## Найденный конфликт окружения (важно для PR4!)

На сервере уже стоит PostgreSQL 15 + PostGIS 3.5, которые регистрируют
**machine-level** переменную окружения `PROJ_LIB`, указывающую на свою
(более старую) `proj.db`:

```
PROJ_LIB = C:\Program Files\PostgreSQL\15\share\contrib\postgis-3.5\proj
```

PDAL из conda-окружения подхватывает эту системную переменную и падает:

```
PROJ: proj_create_from_database: ...proj.db contains DATABASE.LAYOUT.VERSION.MINOR = 2
whereas a number >= 6 is expected. It comes from another PROJ installation.
```

**Решение — НЕ трогать системную `PROJ_LIB`** (это сломало бы существующие
геозапросы PostGIS, которыми пользуется остальное приложение), а
переопределять `PROJ_DATA`/`PROJ_LIB` только для процесса конвертации,
указывая на proj.db самого conda-окружения:

```powershell
$env:PROJ_DATA = 'C:\Users\admin\miniforge3\envs\geo\Library\share\proj'
$env:PROJ_LIB  = $env:PROJ_DATA
```

Без этого шага конвертация LAS→COPC не работает на этом сервере — это
не теоретический риск, а реально воспроизведённая и решённая проблема
(см. ниже, "Проверено вручную").

## Проверенная команда конвертации

```powershell
$env:PROJ_DATA = 'C:\Users\admin\miniforge3\envs\geo\Library\share\proj'
$env:PROJ_LIB  = $env:PROJ_DATA
& C:\Users\admin\miniforge3\envs\geo\Library\bin\pdal.exe translate <input>.las <output>.copc.laz
```

PR4 (LAS-стриминг) должен вызывать `pdal.exe` именно с этими двумя
переменными окружения, установленными в дочернем процессе (не глобально).

## Проверено вручную (на реальном файле с сервера)

Файл `uploads/tours/tour_6a3af515ed1716.67716215_rexf.las` (59 107 818
байт, ~1.97 млн точек, СК МСК-168 Красноярский край):

- Конвертация: 2.8 сек.
- Результат: `.copc.laz`, 9 818 470 байт (~6× меньше за счёт LAZ-сжатия).
- Прочитан обратно через `pdal info --summary` — корректные bounds, SRS
  (WKT/proj4) сохранён без потерь, `num_points` совпадает.

Большие файлы (на сервере есть LAS вплоть до ~8.3 ГБ) не тестировались
из-за длительности — линейная экстраполяция по замеру на 59 МБ даёт
ориентировочно несколько минут на самый крупный файл; точное время на
файлах такого размера будет видно по факту в PR4/при реальном использовании.

## Параллельность (см. docs/CURRENT_STATE.md, раздел 10, п.4)

Сервер физический: Intel i9-10900F (10 ядер/20 потоков), 64 ГБ RAM.
Конвертация в PDAL однопоточная по умолчанию — при нескольких одновременных
запросах на конвертацию запускать не более `floor(20/2) = 10` процессов
параллельно (резерв под Apache/БД/NTRIP-поллинг/RINEX-очередь). Сам
механизм очереди — код PR4, здесь не реализуется.
