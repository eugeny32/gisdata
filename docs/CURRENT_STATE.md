# Текущее состояние PlayCanvas-вьювера (Этап 0 — анализ)

Документ написан ДО какого-либо кода по новому ТЗ. Важная оговорка сразу:
**ТЗ описывает проект на Vite/Webpack/TS с npm-пакетами движка — у нас НЕ так.**
Ниже — фактическое устройство, проверенное по реальным файлам репозитория, и
отдельно — что подтвердилось/не подтвердилось при проверке самого движка
PlayCanvas 2.20.0 (исходники на GitHub, реестр npm, jsdelivr).

## 1. Реальный стек проекта (не Vite/TS!)

- Backend: чистый PHP (без фреймворка), PostgreSQL, без npm/Vite/Webpack/TypeScript
  вообще — ни для backend, ни для фронтенда.
- Карта/2D: Leaflet, подключён через `<script src=...>` с unpkg CDN.
- 3D-вьювер: **PlayCanvas Engine подключается через `<script type="importmap">`
  прямо в HTML** (CDN jsdelivr, `playcanvas@2.20.0/build/playcanvas/src/index.js`),
  без сборки — обычный ES-модуль импортируется `await import('playcanvas')`
  прямо в инлайн-скрипте внутри PHP-шаблона. Никакого `package.json` для
  фронтенда не существует.
- Где живёт код вьювера:
  - `map.php` — основной вьювер тура (внутри Bootstrap-модалки), функция
    `loadTourScenePlayCanvas(urls, modelType)` и её хелперы
    `loadSplatFiles()` / `loadLasFiles()`.
  - `playcanvas_test.php` — изолированная тестовая копия того же кода
    (полностью на весь экран, без модалки) — использовалась для проверки
    гипотез без риска для продакшна.
  - `las_test.php` — более старая тестовая страница на THREE.js (осталась
    от предыдущей итерации LAS-вьювера, до перехода на PlayCanvas).

Вывод: формат поставки "каждый модуль = отдельная ветка/PR с TS-типами"
буквально не применим без отдельного решения по тулчейну (см. раздел 6,
открытый вопрос).

## 2. Инициализация сцены/камеры — сейчас полностью самописная

`loadTourScenePlayCanvas()` в `map.php`:
- `new pc.Application(canvas, {...})`, `FILLMODE_NONE` + свой
  `ResizeObserver` (канвас должен заполнять контейнер модалки, а не всё окно).
- Камера — **ручная орбитальная** (yaw/pitch/distance, свои обработчики
  `pointerdown/pointermove/wheel`), НЕ `OrbitControls`/`CameraControls` из
  движка — их в проекте никогда не подключали.
- Никакого режима Walk/Fly нет вообще.
- Штурвал-гизмо (аналог ViewCube) — **уже реализован вручную**: отдельная
  маленькая `pc.Layer` + вторая камера с `rect`-вьюпортом в углу + куб из
  6 раскрашенных по осям `pc.Entity` с `render: {type:'plane'}` + свой
  ray-box тест по клику для привязки к стандартным видам. Это РАБОТАЕТ и
  проверено живыми скриншотами в этой сессии — модуль 1.2 нового ТЗ не
  с нуля, а доработка существующего.

## 3. Загрузка LAS — сейчас целиком в память, без индексации

- Парсер: `@loaders.gl/core` + `@loaders.gl/las` (`LASLoader`), тоже через
  CDN importmap.
- Файл скачивается ПОЛНОСТЬЮ (`fetch` + `ReadableStream` с прогресс-баром
  по байтам) в один `ArrayBuffer`, затем `parse(buffer, LASLoader)` — нет
  частичной/потоковой загрузки по области, нет октодерева, нет LOD.
- Точки — один `pc.Mesh` на файл (`PRIMITIVE_POINTS`), свой
  `pc.ShaderMaterial` (фиксированный `gl_PointSize`, цвет — атрибут
  `COLOR_0`, либо HSL-заливка по высоте, если в файле нет реального RGB).
  Цвет "запекается" в буфер ОДИН раз при загрузке — переключение режима
  раскраски в реальном времени сейчас невозможно без пересборки буфера.
- Никаких атрибутов intensity/classification/return number не читается и
  не передаётся в шейдер — только POSITION и COLOR_0.

## 4. Загрузка 3DGS — нативный `gsplat`, но сырой `.ply`, без LOD

- `new pc.Asset(name, 'gsplat', {url})` + `entity.addComponent('gsplat', {asset})`
  — это штатный механизм движка (подтверждено: `GSplatComponentSystem` и
  `GSplatHandler` регистрируются автоматически в `new pc.Application`).
- Формат текущих файлов на сервере — **сырой несжатый `.ply`** (например
  `house_lat/*.ply`, `shule_ekb/*.ply`), НЕ SOG и НЕ streamed-LOD-папка.
- Перед сохранением `.ply` уже прогоняется через `@playcanvas/splat-transform`
  (CLI, установлен на сервере, Node.js) — но только для шумоподавления
  (`-N -V scale_*,lt,2 -G`), НЕ для конвертации в SOG/streamed-LOD и НЕ для
  генерации `-K` коллайдера.
- Прогресс загрузки — через `asset.on('progress', ...)`, целиком один файл
  за раз (без LOD, без стриминга по видимой области).
- Известное жёсткое ограничение библиотеки (уже наступили на это в сессии):
  текстура центров/цветов у `gsplat`-пайплайна растёт под общее число
  сплатов; на сканах в десятки-сотни миллионов сплатов это упирается в
  `gl.MAX_TEXTURE_SIZE` устройства. Переход на SOG/streamed-LOD (модуль 2.3
  нового ТЗ) полезен вдвойне — не только "грузить по виду", но и снимает
  этот потолок текстуры.

## 5. UI-слой — чистый Bootstrap/DOM, без PCUI/React

Кнопки в шапке модалки (`#tourViewerModal`), панель слоёв, тулбар рисования
аннотаций — всё на Bootstrap 5 + ванильный JS. Никакого PCUI, никакого
React. Единой "панели настроек" (модуль 6 нового ТЗ) нет вообще.

**Самый важный пункт для решения перед стартом: Walk Mode.** Я не нашёл
подтверждения, что это существующая, готовая к использованию фича движка
или официального вьювера PlayCanvas — ни в исходниках движка, ни в
SuperSplat (где он бы логично жил). Возможно: (а) это очень недавняя фича
(новее моих данных) и просто не видна через GitHub/npm с теми запросами,
которые я делал; (б) она существует только в закрытой части
playcanvas.com (хостинг/Editor), не в открытом движке; (в) бриф описывает
её по памяти/маркетингу неточно. **Прежде чем планировать Walk Mode как
"подключить готовое" — нужен прямой источник (ссылка/туториал), который
вы упомянули — без него планирую этот модуль как "писать с нуля" (как уже
и предписано в ТЗ для LAS), на базе `-K`-коллайдера от `splat-transform`
+ своего contoller'а с raycast/collision против полученного `.glb`.

## 7. Инфраструктура: чего у нас физически нет

- **PDAL / entwine / untwine** (нужны для LAS → COPC/Potree) — не
  установлены на сервере (Windows + XAMPP), это Python/C++ GIS-тулчейн,
  отдельная установка, не пересекается ни с чем, что уже есть.
- Build-пайплайн для фронтенда (Vite/Webpack/TS) — отсутствует полностью.
- `splat-transform` (Node CLI) — **есть**, уже используется (подтверждено,
  не предположение) для денойза `.ply` при сохранении тура.

## 10. Решения (зафиксированы перед стартом)

1. **Сборка**: вводим Vite + TypeScript, но только для кода вьювера —
   отдельная папка `viewer/` в репозитории, собирается в статический бандл
   (`.js`), который кладётся как обычный ассет и подключается из `map.php`
   тем же `<script type="module">`. PHP/Apache-рантайм не меняется, Node
   нужен только на этапе сборки (можно собирать локально и заливать
   результат по SCP — как сейчас остальные файлы).
2. **LAS-стриминг**: PDAL/entwine → COPC. Требует отдельной установки на
   сервере (Windows) — это отдельная инфраструктурная задача ДО кода LAS-
   стриминга, не часть кода приложения.
3. **Walk Mode**: пишем с нуля (подтверждённых готовых исходников/ссылки
   нет) — на базе `-K`-коллайдера от `splat-transform` для сплатов, и
   аналогичного по духу контроллера для LAS (без коллизий, либо с упрощённой
   проверкой по тому же `.glb`, если он сгенерирован).
4. **Загрузка серверного железа под тяжёлые конвертации** (подтверждено
   пользователем — сервер физический: Intel i9-10900F, 10 ядер/20 потоков,
   64 ГБ RAM, Radeon RX Vega; "максимально использовать ресурсы сервера,
   если это разгружает клиента, без ущерба работе ОС"). Применимо к PR3
   (PDAL/entwine LAS→COPC) и PR5 (`splat-transform` SOG/streamed-LOD/`-K`):
   - Обе конвертации — CPU/RAM-bound оффлайн-задачи на сервере, рендер в
     браузере клиента (WebGL) они не разгружают сами по себе — разгружают
     именно за счёт того, что КЛИЕНТ получает уже подготовленные
     LOD/COPC-данные вместо сырого файла целиком (меньше скачивать/парсить
     в браузере). Серверный GPU (Radeon RX Vega) для самого рендера тура
     не участвует и не может — рендер архитектурно всегда в браузере
     клиента; GPU сервера не планируется использовать.
   - Конвертации запускаются как пул воркеров с ограничением одновременных
     задач (не больше `floor(NumberOfLogicalProcessors / 2)` = 10 на этом
     сервере, не все 20 потоков — резерв под Apache/БД/NTRIP-поллинг/RINEX-
     очередь, которые уже постоянно работают на той же машине), а не один
     процесс за раз и не неограниченно параллельно.
   - Лимит конфигурируется (env/конфиг), а не хардкодится — на случай
     переезда на другое железо.
   - Решение по конкретной реализации пула (что использовать — Windows
     Task Scheduler с несколькими параллельными задачами с обоюдным
     взаимным excluding через `SELECT ... FOR UPDATE SKIP LOCKED` по
     аналогии с уже работающей очередью `bin/process_rinex_requests.php`,
     либо отдельный долгоживущий воркер-процесс) — откладывается до самого
     PR3/PR5, когда появится код, который нужно параллелить; сейчас
     зафиксировано только ограничение и его обоснование.

## 12. PR4 — найденные живыми тестами баги (важно для PR5/PR6, тот же loader)

1. **Относительный URL → файловая ветка copc.js.** `Getter.create()` в
   `copc` определяет HTTP по префиксу `http(s)://`; наш URL вида
   `/uploads/...` уходил в файловую ветку (`fs.promises.access`), которой
   в браузере нет. Фикс: `new URL(url, window.location.origin)` перед
   любым вызовом `Copc.*`.
2. **`base` в `vite.config.ts`.** Без него Vite генерирует URL воркера
   как абсолютный от корня сайта (`/assets/copcWorker-*.js`), а бандл
   реально лежит в `/assets/viewer/`. Добавлено `base: '/assets/viewer/'`.
3. **`laz-perf` web-сборка не работает в Worker.** У пакета отдельная
   сборка `laz-perf/lib/worker/index.js` — обычный (`browser`-field)
   вход рассчитан на DOM и падает в Worker ("both async and sync
   fetching of the wasm failed"). Нужно импортировать worker-сборку
   явно и передавать готовый `lazPerf` в `Copc.loadPointDataView(...,
   { lazPerf })`, а сам `.wasm` тянуть через `?url`-импорт Vite (иначе
   относительный путь `laz-perf.wasm` в самом Emscripten-лоадере 404
   возвращает HTML, а не WASM).
4. **Центр padded-куба ≠ центр данных.** `copc.info.cube` — корень
   octree, дополненный до правильного куба; на плоских объектах одна из
   осей куба может быть растянута в разы относительно реальных точек
   (здание 434×434×52м получило куб 434×434×434м). Центрировать камеру
   нужно по `copc.header.min/max` (реальные данные), а не по центру
   `cube` — иначе геометрия оказывается далеко от того, куда смотрит
   камера, при этом узлы исправно проходят фрустум-тест и грузятся.
5. **Бюджет точек проверялся неверно при диспетчеризации.** `budgetUsed`
   из обхода дерева — это сумма по ВСЕМ найденным узлам сразу; сравнение
   `budgetUsed <= POINT_BUDGET` на каждой итерации цикла отправки запросов
   либо пропускало все узлы целиком (если сумма больше бюджета — а это
   обычный случай для богатой сцены), либо не ограничивало ничего.
   Нужен отдельный счётчик, растущий по факту реальной отправки запросов,
   независимый от суммы по дереву обхода.
