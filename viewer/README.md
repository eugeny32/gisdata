# Tour Viewer (PR0 — сборка)

PlayCanvas-вьювер тура (3DGS-сплаты + LAS-облака точек), вынесенный из
инлайн-скрипта `map.php` в TypeScript-модули. Это **единственная** часть
проекта, которая собирается через Vite — весь остальной gisdata остаётся
чистым PHP без сборки.

## Почему так

`map.php` подключал PlayCanvas через `<script type="importmap">` прямо в
HTML и писал весь код вьювера одним большим инлайн-скриптом. Это работало,
но плохо масштабируется на модульное ТЗ (навигация/стриминг/раскраска/
сечения как отдельные модули с типами). Решение — отдельный TS-проект,
который собирается в один статический `.js`-файл и подключается как
обычный ассет, без изменения остального стека (PHP/Apache/без Node в
рантайме продакшна).

## Сборка и деплой

```
cd viewer
npm install
npm run build
```

Результат — `../assets/viewer/tour-viewer.js` (commit-нутый в репозиторий,
как обычный статический файл — на сервере Node не нужен, сборка делается
локально/на CI, на сервер просто заливается готовый `.js`, как и весь
остальной проект — SCP/FTP, см. `bin/deploy_ftp.ps1`).

## Внешние зависимости — НЕ бандлятся

`playcanvas`, `@loaders.gl/core`, `@loaders.gl/las` помечены `external` в
`vite.config.ts` — в собранном файле остаются как обычные ES-импорты и
резолвятся в браузере через `<script type="importmap">`, который уже есть
в `app/views/_head.php` (CDN jsdelivr). Это сделано намеренно:

- не раздувать бандл копией движка;
- не дублировать версию движка в двух местах (CDN importmap — единственный
  источник истины по версии playcanvas).

`playcanvas` присутствует в `devDependencies` только ради типов при сборке
— `npm install` не подключается к продакшну.

## Публичный API (`window.TourViewer`)

`map.php` обращается к вьюверу только через этот объект — никаких прямых
ссылок на внутренние переменные модуля больше нет:

- `TourViewer.load(urls, modelType)` — открыть тур.
- `TourViewer.dispose()` — закрыть/освободить текущий вьювер (модалка
  закрыта).
- `TourViewer.recenter()` — кнопка «Центрировать» в шапке модалки.
- `TourViewer.showError(message)` / `TourViewer.hideError()` — оверлей
  ошибки загрузки.

## Что НЕ менялось в этом PR

Это перенос 1:1 — поведение идентично тому, что было в инлайн-скрипте
`map.php` до PR0 (включая известные ограничения: LAS грузится целиком в
память, 3DGS — сырой `.ply` без LOD, штурвал-гизмо — тот же ray-box тест по
6 граням). Архитектурные улучшения (переиспользуемый ViewCube,
Orbit/Fly/Walk как явные классы, стриминг, раскраска, сечения) —
последующие PR, см. `docs/CURRENT_STATE.md`, раздел 11.

## Структура

- `src/types.ts` — общие типы (`ModelType`, `PcModule`).
- `src/constants.ts` — `AXIS_FIX_ROTATION`.
- `src/gizmo.ts` — штурвал-кубик (`createNavCubeGizmo`).
- `src/splatLoader.ts` — загрузка 3DGS (`loadSplatFiles`).
- `src/lasLoader.ts` — загрузка LAS (`loadLasFiles`).
- `src/tourViewer.ts` — оркестрация: камера, канвас, жизненный цикл
  `pc.Application` (`loadTourScene`/`disposeTourViewer`/`recenterTourCamera`).
- `src/main.ts` — точка входа, формирует `window.TourViewer`.

## Ручной чек-лист приёмки PR0

      ориентация корректна, штурвал работает.
- [ ] Открытие тура с LAS — точки отображаются, прогресс по байтам/разбору
      идёт, штурвал работает.
- [ ] Кнопка «Центрировать» работает для обоих типов.
- [ ] Повторное открытие/закрытие модалки несколько раз подряд — нет утечек
      канваса/дублирования рендер-циклов (см. `dispose()`).
- [ ] Ошибка загрузки (например, несуществующий файл) показывает оверлей
      с текстом ошибки.
